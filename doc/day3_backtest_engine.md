# Day 3：回测框架搭建

> 没有回测，量化策略就是纸上谈兵。今天我们搭建一个**工程化的回测引擎**，
> 它将成为后续所有策略开发的基座。

---

## 一、为什么需要回测框架？

回看 Day 1 的代码：

```python
df["Signal"] = 0
df.loc[df["SMA_20"] > df["SMA_50"], "Signal"] = 1
df["Strategy_Return"] = df["Daily_Return"] * df["Signal"].shift(1)
```

这段代码有几个**严重问题**：

1. **没有交易费用**：A 股有佣金（~0.025%）和印花税（卖出 0.05%），频繁交易会被费用吃掉
2. **信号和回测耦合**：策略逻辑和回测计算混在一起，换个策略就得重写
3. **没有仓位管理**：只有 0（空仓）和 1（满仓），没有分仓概念
4. **没有交易记录**：不知道什么时候买卖了，每笔赚了多少
5. **不可复用**：每写一个新策略，都要重复写回测代码

**解决方案**：用面向对象（OOP）搭建一个可复用的回测框架。

## 三、A 股交易费用详解

这是量化回测中**最容易被忽略但影响巨大**的因素。

### 3.1 费用构成

| 费用项 | 比例 | 方向 | 说明 |
| ------ | ---- | ---- | ---- |
| 佣金 | ~0.025% | 买 + 卖（双向） | 券商收取，最低 5 元/笔 |
| 印花税 | 0.05% | 仅卖出 | 国家收取，2023年减半 |
| 过户费 | 0.001% | 买 + 卖 | 中国结算收取，可忽略 |

### 3.2 费用影响有多大？

假设一个策略年换手率 50 倍（每月换仓约 4 次）：

$$
\text{年交易成本} = 50 \times (0.025\% \times 2 + 0.05\%) = 50 \times 0.1\% = 5\%
$$

**年化 5% 的交易成本！** 这意味着你的策略年化收益必须超过 5% 才能覆盖费用。

> [!WARNING]
> 很多看起来"很赚钱"的策略，扣除交易费用后就不赚钱了。
> 这就是为什么回测**必须**加入交易费用！

### 3.3 滑点 (Slippage)

**滑点** = 你期望的成交价格和实际成交价格之间的差异。

原因：
- 你发出买入指令时，价格可能已经变了
- 大单会推动价格（市场冲击）

在回测中，我们通常假设一个固定的滑点比例（如 0.01%~0.05%）。

## 四、关键概念：信号 → 仓位 → 收益

这是回测中最核心的数据流：

```
Signal[t]  →  Position[t]  →  Return[t+1]
 (策略判断)    (今天持仓)      (明天结算)
```

### 4.1 为什么用 `shift(1)`？

```python
df["Strategy_Return"] = df["Daily_Return"] * df["Position"].shift(1)
```

**关键**：今天收盘后产生的信号，只能在**明天**执行。因为：
- 信号基于今天的收盘价计算
- 你在今天收盘后才能看到信号
- 最早要到明天开盘才能交易

这叫做**避免未来函数（look-ahead bias）**，是回测中最常犯的错误之一。

### 4.2 什么是交易信号变化？

```
Day 1: Position = 0 (空仓)
Day 2: Position = 1 (买入) ← 这里产生了一笔"买入"交易
Day 3: Position = 1 (持有) ← 没有交易
Day 4: Position = 0 (卖出) ← 这里产生了一笔"卖出"交易
```

只有仓位**发生变化**时才产生交易，才需要扣除交易费用。

## 五、代码结构说明

今天的代码文件 `day3_backtest_engine.py` 包含：

### 5.1 `Strategy` 抽象基类

```python
class Strategy(ABC):
    @abstractmethod
    def generate_signals(self, df: pd.DataFrame) -> pd.Series:
        """返回信号序列：1=买入/持有, 0=空仓, -1=做空(A股暂不支持)"""
        pass
```

每个新策略只需继承 `Strategy` 并实现 `generate_signals()` 方法。

### 5.2 `BacktestEngine` 回测引擎

核心方法：
- `run()`：主流程——信号生成 → 仓位管理 → 费用计算 → 绩效统计
- `_calculate_costs()`：根据仓位变化计算交易费用
- `_calculate_metrics()`：Day 2 学过的所有风险指标
- `plot_report()`：生成绩效报告图表

### 5.3 `SMAStrategy` 示例策略

用 Day 1 的双均线策略作为演示，但这次通过框架运行，有费用、有交易记录！

## 六、回测引擎的运行流程

```
1. 加载数据 (DataFrame with OHLCV)
         │
         ▼
2. 策略生成信号 (Strategy.generate_signals)
    Signal: [0, 0, 1, 1, 1, 0, 0, 1, ...]
         │
         ▼
3. 信号延迟一天 (shift(1)，避免未来函数)
    Position: [NaN, 0, 0, 1, 1, 1, 0, 0, ...]
         │
         ▼
4. 检测交易（仓位发生变化的时刻）
    Trade: [-, -, -, BUY, -, -, SELL, -, ...]
         │
         ▼
5. 计算交易费用（佣金 + 印花税 + 滑点）
    Cost: [0, 0, 0, 0.075%, 0, 0, 0.1%, 0, ...]
         │
         ▼
6. 计算策略收益
    Strategy_Return = Position × Daily_Return - Cost
         │
         ▼
7. 计算绩效指标 (Sharpe, MaxDD, Sortino, ...)
         │
         ▼
8. 生成报告 (图表 + 打印)
```

## 七、思考题

1. **未来函数**：如果我们不用 `shift(1)`，回测结果会怎样？试着去掉 shift 看看 Sharpe 会变成多少。这就是"作弊"回测。

2. **交易频率与费用的关系**：修改 SMA 的窗口参数（比如 SMA_5 和 SMA_10），观察交易次数和费用如何变化。高频策略的费用问题有多严重？

3. **（进阶）最小佣金的影响**：现在我们的代码没有实现"最低 5 元佣金"的逻辑。如果你管理的资金只有 1 万元，每次交易金额很小，最低佣金的影响有多大？
